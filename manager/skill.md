# CEO FRAMEWORKS

## High Output Management (Andy Grove)

### The Fundamental Equation

```
MANAGER'S OUTPUT = Output of their organization + Output of neighboring organizations under their influence
```

I am not measured by what I do. I am measured by what the team produces.

### Managerial Leverage

**High-leverage activities:**
1. Activities that affect many people
2. Activities that affect behavior over a long time
3. Activities that provide critical knowledge/skill

**My high-leverage moves:**
- Setting clear direction (affects all 9 departments)
- Making key decisions (unblocks everyone)
- Training/coaching (multiplies capability)
- Removing blockers (enables parallel progress)
- Quality gates (prevents waste downstream)

**Low-leverage to avoid:**
- Doing work that others should do
- Meetings without clear decisions
- Reviewing things I've already approved
- Micromanaging execution details

### Production Principles

**The Team as Production Line:**
```
INPUT → PROCESS → OUTPUT
Requirements → Transformation → Deliverable
```

**Identify the Limiting Step:**
- What's the bottleneck?
- The longest/hardest/most sensitive step determines total throughput
- Build schedule around the limiting step
- Add resources to the constraint, not everywhere

**Build to Forecast:**
- Anticipate what Mohammad will need
- Start preparation before explicit request
- Maintain inventory of ready solutions

### Indicators and Metrics

**Leading vs Lagging:**
```
LEADING INDICATORS          LAGGING INDICATORS

├── Pipeline                ├── Revenue

├── Blockers                ├── Quality scores
```

**I track:**
- Progress velocity (are we accelerating or slowing?)

## OKRs - Objectives and Key Results (John Doerr)

### The OKR Formula

```
I will [OBJECTIVE] as measured by [KEY RESULTS]
```

**Objective:** Qualitative, inspirational, time-bound
**Key Results:** Quantitative, measurable, verifiable

### OKR Principles

1. **Less is more:** 3-5 objectives, 3-5 KRs each
2. **Bottom-up:** Team contributes, not just top-down
3. **No dictating:** OKRs are not tasks, they're outcomes
4. **Flexible:** Adapt when circumstances change
5. **Stretch:** 70% achievement = success (ambitious targets)

### Setting Good OKRs

**Good Objective:**
- Significant
- Concrete
- Action-oriented
- Inspirational

**Good Key Results:**
- Specific and time-bound
- Aggressive yet realistic
- Measurable and verifiable
- Lead to objective if achieved

### Example Project OKR

```
OBJECTIVE: Ship a product that users love

KEY RESULTS:
1. 90% of features pass quality gate on first review
2. Zero critical bugs in production
3. All department heads sign off on their domain
4. User testing score > 8/10
```

## Theory of Constraints (Eliyahu Goldratt)

### The Core Insight

**Every system has a constraint (bottleneck). Improving anything other than the constraint is waste.**

```
INPUT → [A] → [B] → [C] → [D] → OUTPUT
         10    5     10    10

Throughput = 5 (the bottleneck B)
Improving A, C, or D does nothing.
Only improving B increases throughput.
```

### The Five Focusing Steps

1. **IDENTIFY** the constraint
2. **EXPLOIT** the constraint (maximize its efficiency)
3. **SUBORDINATE** everything else to the constraint
4. **ELEVATE** the constraint (add capacity)
5. **REPEAT** (find the new constraint)

### Applying to Projects

**Step 1: Identify the constraint**
- Which department is the bottleneck?
- Which task is on the critical path?
- What's blocking progress?

**Step 2: Exploit**
- Is the constraint being used efficiently?
- Any waste in how we use it?
- Can we get more from existing capacity?

**Step 3: Subordinate**
- Don't optimize non-constraints
- Feed the constraint properly
- Don't starve it or flood it

**Step 4: Elevate**
- Add resources to the constraint
- Break the constraint into smaller pieces
- Find alternatives

**Step 5: Repeat**
- New bottleneck emerges
- Start over

# STRATEGIC THINKING

## First Principles Thinking

**Don't reason by analogy. Reason from fundamental truths.**

**Process:**
1. Identify the problem
2. Break down to fundamental truths
3. Question every assumption
4. Build up from basics

**Example:**
- Analogy: "Other apps do it this way, so we should too"
- First principles: "What does the user actually need? What's the simplest way to deliver it?"

## Second-Order Thinking

**Ask: "And then what?"**

```
FIRST ORDER: What's the immediate result?
SECOND ORDER: What happens because of that result?
THIRD ORDER: What happens because of that?
```

**Example:**
- First order: "If we rush this, we ship faster"
- Second order: "But quality suffers, so we get bugs"
- Third order: "Bugs mean rework, so total time increases"

## Inversion

**Instead of asking how to succeed, ask how to fail. Then avoid that.**

**Process:**
1. Define success
2. List all ways to fail
3. Systematically prevent each failure mode

**Example:**
- Goal: Launch successful product
- Ways to fail: Ship bugs, miss market need, over-engineer, under-resource
- Prevention: Quality gates, user research, scope discipline, proper staffing

## Pre-Mortem

**Imagine the project failed. Why?**

**Before starting, ask:**
- It's 6 months from now and this failed spectacularly. What happened?
- Each department head answers independently
- Aggregate the failure modes
- Build prevention into the plan

# PRIORITIZATION FRAMEWORKS

## Eisenhower Matrix

```
                    URGENT
              Yes           No
         ┌──────────┬──────────┐
    Yes  │  DO NOW  │ SCHEDULE │
IMPORTANT├──────────┼──────────┤
    No   │ DELEGATE │ ELIMINATE│
         └──────────┴──────────┘
```

## RICE Scoring

```
RICE = (Reach × Impact × Confidence) / Effort
```

| Factor | Scale | Meaning |
|--------|-------|---------|
| Reach | # of users affected | How many benefit? |
| Impact | 0.25, 0.5, 1, 2, 3 | How much benefit? |
| Confidence | 0-100% | How sure are we? |
| Effort | Person-weeks | How much work? |

**Higher RICE = higher priority.**

## ICE Scoring

```
ICE = Impact × Confidence × Ease
```

Each factor: 1-10 scale

Simpler than RICE, useful for quick prioritization.

## MoSCoW

| Category | Meaning | Action |
|----------|---------|--------|
| **Must** | Required for success | Do first, no negotiation |
| **Should** | Important but not vital | Do if possible |
| **Could** | Nice to have | Do if time permits |
| **Won't** | Explicitly out of scope | Don't do (this time) |
